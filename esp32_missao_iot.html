O QUE É O ESP32?

O ESP32 é um microcontrolador de duplo núcleo desenvolvido pela Espressif Systems (China). Lançado em 2016, tornou-se o chip mais popular do mundo para projetos IoT por uma razão simples: ele tem Wi-Fi + Bluetooth integrados no próprio chip — algo que antes exigia módulos externos caros.

No contexto da disciplina de 2 ADS-AMS, o ESP32 é a porta de entrada para a integração com dispositivos IoT, capturando dados de sensores que alimentam seus sistemas distribuídos com backends, brokers MQTT e data streaming.

Analogia: pense no ESP32 como um computador minúsculo — ele tem processador, memória, conectividade de rede e pinos para interagir com o mundo físico, tudo em um chip do tamanho de uma moeda de R$0,50.

GPIO em Profundidade

// digital, analógico, PWM, interrupções e modos de operação

1. GPIO DIGITAL — SAÍDA E ENTRADA

GPIO significa General Purpose Input/Output — pino de propósito geral que pode ser configurado como entrada ou saída digital. Opera com lógica 3.3V (HIGH = 3.3V, LOW = 0V).

OUTPUT (Saída)

Controla LEDs, relés, buzzers, motores (via transistor). O ESP32 fornece corrente para o circuito externo. Max: 40mA por pino, 1200mA total.

INPUT (Entrada)

Lê botões, sensores digitais (PIR, ultrassom), chaves. Pode ter resistor pull-up ou pull-down interno ativado por software.

INPUT_PULLUP

Resistor interno de ~45kΩ ligado ao 3.3V. Botão conectado ao GND — lê HIGH em repouso, LOW quando pressionado.

INPUT_PULLDOWN

Resistor interno ligado ao GND. Lê LOW em repouso, HIGH quando ativo. Menos comum, mas disponível.

2. ADC — CONVERSOR ANALÓGICO-DIGITAL

O ADC (Analog-to-Digital Converter) converte tensão analógica (0–3.3V) em número inteiro. Com resolução de 12 bits, temos 4096 valores possíveis (0 a 4095).

Fórmula: Tensão = (leitura / 4095) × 3.3V  |  Leitura = (tensão / 3.3) × 4095
CanalPinosDisponível com Wi-Fi?Uso recomendado
ADC1GPIO 32–39SIM ✅Sensores analógicos em projetos IoT
ADC2GPIO 0,2,4,12–15,25–27NÃO ⚠️Apenas sem Wi-Fi ativo — evitar em IoT
Problema do ADC2: quando o Wi-Fi está ativo, o ADC2 fica inutilizável — a antena usa os mesmos circuitos. Para projetos IoT, use sempre ADC1 (GPIO 32–39).
3. PWM — MODULAÇÃO POR LARGURA DE PULSO

PWM simula uma saída analógica usando pulsos digitais rápidos. O ESP32 tem um controlador LEDC (LED Control) com 16 canais PWM independentes, podendo usar qualquer GPIO como saída PWM.

Duty Cycle

% do tempo em HIGH. 0% = sempre LOW, 50% = metade do tempo HIGH, 100% = sempre HIGH. Controla brilho de LED.

Frequência

Velocidade dos pulsos. LEDs: 1-5 kHz. Servos: 50 Hz. Motores DC: 1-20 kHz. Configura por canal.

Resolução

Bits de precisão do duty. 8-bit = 256 níveis. 16-bit = 65536 níveis. Maior resolução = controle mais fino.

Fade / Respiração

Hardware fade — o ESP32 altera o duty automaticamente sem ocupar CPU. Perfeito para LEDs "respirando".

4. INTERRUPÇÕES (ISR — Interrupt Service Routine)

Em vez de ficar verificando um pino em loop, você registra uma interrupção: quando o pino muda de estado, o ESP32 para o que está fazendo e chama sua função automaticamente.

RISING

Dispara na borda de subida: quando o pino vai de LOW → HIGH. Ex: botão liberado com pull-down.

FALLING

Dispara na borda de descida: LOW → HIGH → LOW. Ex: botão pressionado com pull-up.

CHANGE

Dispara em qualquer mudança. Ex: encoder rotativo, que muda estado continuamente.

IRAM_ATTR

Obrigatório no ESP32: funções ISR devem ter este atributo — armazena na RAM interna para acesso rápido.

Regra ISR: mantenha a função de interrupção ultra-rápida. Não use Serial.print(), delay(), ou funções bloqueantes dentro de uma ISR. Use flags e processe no loop().
5. TOUCH — SENSOR CAPACITIVO SEM HARDWARE

O ESP32 tem 10 pinos touch capacitivos — detectam presença do dedo sem nenhum componente adicional. Internamente medem a capacitância do pino: quando você toca, a capacitância aumenta.

Protocolos de Comunicação

// UART · I2C · SPI · Wi-Fi · BLE · MQTT — conectando o ESP32 ao mundo

UART — COMUNICAÇÃO SERIAL ASSÍNCRONA

O protocolo mais simples: dois fios (TX e RX), sem clock compartilhado. O ESP32 tem 3 UARTs.

UARTTX / RX padrãoUso típico
UART0GPIO 1 / GPIO 3Debug serial (USB) — evitar para outros usos
UART1GPIO 10 / GPIO 9Flash SPI — remapeável para outros GPIOs
UART2GPIO 17 / GPIO 16Sensores: GPS, GSM, módulos externos
Velocidades comuns (baud rate): 9600, 115200, 921600 bps. Arduino IDE usa 115200 por padrão. GPS usa 9600. ESP-IDF logging usa 115200.
I2C — BARRAMENTO SERIAL SÍNCRONO (2 FIOS)

I2C usa apenas 2 fios (SDA + SCL) e suporta múltiplos dispositivos no mesmo barramento, cada um com endereço único de 7 bits (até 127 dispositivos!).

SDA (Data)

Linha de dados bidirecional. GPIO 21 por padrão. Pull-up de 4.7kΩ necessário.

SCL (Clock)

Sinal de sincronismo gerado pelo master. GPIO 22 por padrão. Pull-up de 4.7kΩ necessário.

Velocidades

Standard: 100kHz. Fast: 400kHz. Fast+: 1MHz. Suficiente para a maioria dos sensores.

Sensores I2C comuns

DHT — temperatura/umidade. MPU6050 — acelerômetro. BMP280 — pressão. OLED SSD1306. RTC DS3231.

Endereços: cada dispositivo I2C tem endereço fixo (ex: BMP280 = 0x76 ou 0x77). Use o I2C Scanner para descobrir dispositivos conectados.
SPI — BARRAMENTO SERIAL DE ALTA VELOCIDADE (4 FIOS)

SPI é mais rápido que I2C mas usa 4 fios e suporta múltiplos dispositivos via pinos CS separados. Ideal para comunicação de alta taxa.

SinalFunçãoGPIO padrão
MOSIMaster Out, Slave In (dados do ESP para dispositivo)GPIO 23
MISOMaster In, Slave Out (dados do dispositivo para ESP)GPIO 19
SCLKClock (sincronismo)GPIO 18
CS/SSChip Select (seleciona qual dispositivo falar)GPIO 5 (ou qualquer GPIO)
Quando usar SPI? Cartões SD (leitura de arquivos), displays TFT/ILI9341, módulos RF (nRF24L01, LoRa), leitor RFID/NFC (MFRC522).
Wi-Fi + MQTT — A CONEXÃO IOT

O ESP32 suporta Wi-Fi nativo e o protocolo MQTT (Message Queuing Telemetry Transport) é o padrão da indústria para IoT — leve, eficiente, funciona em redes instáveis.

Station (STA)

ESP32 conecta à sua rede Wi-Fi. Modo mais comum — acessa internet, MQTT broker, APIs REST.

Access Point (AP)

ESP32 cria sua própria rede. Útil para configuração inicial sem roteador.

STA + AP

Modo simultâneo — conectado à internet E criando hotspot local. Útil para gateways IoT.

MQTT Broker

Servidor de mensagens (Mosquitto, HiveMQ, broker público). ESP32 publica dados; backend Python/Node assina.

Fluxo IoT completo: Sensor → ESP32 → Wi-Fi → MQTT Broker (Mosquitto) → Backend Python → Banco de Dados → Dashboard Web. Exatamente o que a ementa da disciplina pede!
DEEP SLEEP — OTIMIZAÇÃO PARA BATERIA

Para projetos alimentados por bateria, o Deep Sleep é fundamental: o ESP32 consome apenas 10 µA dormindo, contra 240 mA acordado com Wi-Fi.

Timer Wakeup

Acorda após X microssegundos. Ex: sensor que envia temperatura a cada 5 minutos.

GPIO Wakeup

Acorda quando pino muda de estado. Ex: sensor de movimento que só acorda com detecção.

Touch Wakeup

Acorda ao tocar pino capacitivo. Ex: dispositivo wearable que acorda com toque.

ULP Wakeup

Co-processador ULP monitora sensores e acorda o chip principal apenas se necessário.

🔵 DESAFIO 1 — Blink Avançado com Millis()
FÁCIL · +30 XP

O "Hello World" do hardware. Pisque o LED interno sem usar delay() — use millis() para não bloquear o loop e poder fazer outras tarefas simultaneamente.

Por que evitar delay()?

delay(1000) trava o ESP32 por 1 segundo inteiro — nenhum sensor lê, nenhuma comunicação acontece.

millis() retorna o tempo desde o boot (ms) — você checa se passou tempo suficiente sem bloquear.

💡 pinMode(LED_PIN, OUTPUT) para saída. digitalWrite(LED_PIN, ledEstado) para escrever. ledEstado é booleano: true = HIGH, false = LOW.
// ✅ SOLUÇÃO — Blink sem delay() com millis() const int LED_PIN = 2; const long INTERVALO = 500; unsigned long ultimoTempo = 0; bool ledEstado = false; void setup() { Serial.begin(115200); pinMode(LED_PIN, OUTPUT); Serial.println("ESP32 iniciado — Blink sem delay!"); } void loop() { unsigned long agora = millis(); if (agora - ultimoTempo >= INTERVALO) { ultimoTempo = agora; ledEstado = !ledEstado; digitalWrite(LED_PIN, ledEstado); Serial.printf("LED: %s\n", ledEstado ? "ON" : "OFF"); } // Aqui: leia sensores, processe MQTT, etc. sem travar! }
🟡 DESAFIO 2 — Leitura de Sensor Analógico com ADC
MÉDIO · +40 XP

Leia um potenciômetro (ou sensor LM35) via ADC1 e converta para tensão real. Exiba no Monitor Serial e mapeie o valor para controlar o brilho de um LED via PWM.

Hardware simulado: potenciômetro no GPIO 34 (ADC1_CH6)

LED PWM: GPIO 5, canal LEDC 0, frequência 5kHz, resolução 8-bit

Conversão: ADC lê 0–4095 → mapear para 0–255 para PWM → calcular tensão em V

💡 ledcSetup(canal, freq, bits) configura o canal PWM. Tensão máxima no ESP32 = 3.3V. Para map(): o primeiro valor mapeado é 0 (LED apagado quando ADC = 0).
// ✅ SOLUÇÃO — ADC + PWM completo const int ADC_PIN = 34; const int LED_PIN = 5; const int PWM_CANAL = 0; const int PWM_FREQ = 5000; const int PWM_BITS = 8; void setup() { Serial.begin(115200); ledcSetup(PWM_CANAL, PWM_FREQ, PWM_BITS); // 8 bits = 0-255 ledcAttachPin(LED_PIN, PWM_CANAL); Serial.println("ADC + PWM prontos!"); } void loop() { int leitura = analogRead(ADC_PIN); float tensao = (leitura / 4095.0) * 3.3; int brilho = map(leitura, 0, 4095, 0, 255); ledcWrite(PWM_CANAL, brilho); Serial.printf("ADC: %4d | Tensão: %.3fV | PWM: %3d\n", leitura, tensao, brilho); delay(100); }
🔴 DESAFIO 3 — Publicar dados no MQTT via Wi-Fi
DIFÍCIL · +50 XP

O coração do projeto IoT da disciplina: conectar o ESP32 à rede Wi-Fi e publicar dados de sensor em um broker MQTT. Use a biblioteca PubSubClient.

Instale no Arduino IDE / PlatformIO: PubSubClient by Nick O'Leary

Broker público para testes: broker.hivemq.com (porta 1883)

Tópico sugerido: 2 ADS-AMS/esp32/temperatura

Fluxo: conecta Wi-Fi → conecta MQTT → lê sensor → publica JSON → aguarda 5s → repete

💡 WiFi.begin(SSID, SENHA) para conectar. mqttClient.setServer(BROKER, PORTA) para configurar. O payload JSON: {"temp": 25.3, "device": "ESP32"}. Use snprintf(payload, sizeof(payload), "{\"temp\":%.1f}", temperatura).
// ✅ SOLUÇÃO — ESP32 + Wi-Fi + MQTT completo #include <WiFi.h> #include <PubSubClient.h> const char* SSID = "SuaRede"; const char* SENHA = "SuaSenha"; const char* BROKER = "broker.hivemq.com"; const int PORTA = 1883; const char* TOPICO = "2 ADS-AMS/esp32/temperatura"; WiFiClient wifiClient; PubSubClient mqttClient(wifiClient); unsigned long ultimoEnvio = 0; void conectarWiFi() { Serial.printf("\nConectando a %s", SSID); WiFi.begin(SSID, SENHA); while (WiFi.status() != WL_CONNECTED) { delay(500); Serial.print("."); } Serial.printf("\nConectado! IP: %s\n", WiFi.localIP().toString().c_str()); } void conectarMQTT() { while (!mqttClient.connected()) { Serial.print("Conectando MQTT..."); if (mqttClient.connect("ESP32_2 ADS-AMS_2026")) { Serial.println(" conectado!"); // Assinar tópico de comandos (opcional): mqttClient.subscribe("2 ADS-AMS/esp32/cmd"); } else { Serial.printf(" falha rc=%d\n", mqttClient.state()); delay(2000); } } } void setup() { Serial.begin(115200); conectarWiFi(); mqttClient.setServer(BROKER, PORTA); } void loop() { if (!mqttClient.connected()) conectarMQTT(); mqttClient.loop(); // manter conexão viva if (millis() - ultimoEnvio >= 5000) { ultimoEnvio = millis(); float temperatura = random(200, 350) / 10.0; // substitua por sensor real char payload[128]; snprintf(payload, sizeof(payload), "{\"device\":\"ESP32\",\"temp\":%.1f,\"ts\":%lu}", temperatura, millis()); bool ok = mqttClient.publish(TOPICO, payload); Serial.printf("Publicado [%s]: %s (%s)\n", TOPICO, payload, ok ? "OK" : "FALHA"); } }
⚡ DESAFIO 4 — Deep Sleep com envio MQTT
EXPERT · +60 XP

Implemente o ciclo completo de um nó IoT a bateria: acorda do deep sleep, lê sensor, conecta Wi-Fi, publica MQTT, volta a dormir. Use RTC Memory para contar ciclos.

RTC_DATA_ATTR: variáveis com esse atributo sobrevivem ao deep sleep

Ciclo: boot → setup() → publica → esp_deep_sleep_start() → repete após SLEEP_US

Dica: No deep sleep, não há loop() — tudo vai no setup()

💡 Configure o timer ANTES de dormir: esp_sleep_enable_timer_wakeup(SLEEP_US). Depois chame esp_deep_sleep_start(). WiFi.disconnect(true) e delay(100) antes de dormir garante desconexão limpa.
// ✅ SOLUÇÃO — Deep Sleep + MQTT completo (modo produção IoT) #include <WiFi.h> #include <PubSubClient.h> #define SLEEP_US (5 * 60 * 1000000ULL) const char* SSID = "SuaRede"; const char* SENHA = "SuaSenha"; const char* BROKER = "broker.hivemq.com"; const char* TOPICO = "2 ADS-AMS/esp32/dados"; RTC_DATA_ATTR int numeroCiclos = 0; // persiste no RTC memory WiFiClient wifiClient; PubSubClient mqttClient(wifiClient); bool conectar() { WiFi.begin(SSID, SENHA); int tentativas = 0; while (WiFi.status() != WL_CONNECTED && tentativas < 20) { delay(500); tentativas++; } if (WiFi.status() != WL_CONNECTED) return false; mqttClient.setServer(BROKER, 1883); return mqttClient.connect("ESP32_DeepSleep"); } void setup() { Serial.begin(115200); numeroCiclos++; Serial.printf("\n=== CICLO #%d ===\n", numeroCiclos); float temperatura = random(200, 350) / 10.0; float umidade = random(400, 900) / 10.0; if (conectar()) { char payload[200]; snprintf(payload, sizeof(payload), "{\"ciclo\":%d,\"temp\":%.1f,\"umidade\":%.1f,\"uptime\":%lu}", numeroCiclos, temperatura, umidade, millis()); mqttClient.publish(TOPICO, payload, true); // retain=true Serial.printf("Publicado: %s\n", payload); delay(500); // garante envio antes de dormir } else { Serial.println("Falha na conexão — dormindo assim mesmo"); } WiFi.disconnect(true); WiFi.mode(WIFI_OFF); esp_sleep_enable_timer_wakeup(SLEEP_US); Serial.printf("Dormindo por 5 minutos...\nConsumo: ~10µA\n"); delay(100); esp_deep_sleep_start(); } void loop() { /* nunca chamado no deep sleep mode */ }
